Today I learnt

Christoph told us about the Interface Segregation Principle or ISP, which is about breaking interfaces down when they start to do too many things, like having more than about 20 methods that are unrelated.

For example, simplifying a lot, if you had a Vehicle interface with methods like turnRadioOn(), turnLightsOn() and turnEngineOn(), you would probably be fine extracting those into their own interfaces Radio, Lights and Engine. If something changed in the lights department, the radio department shouldn't have to care. Extrapolating this to Java, if these where in jar files, we wouldn't have to recompile and deploy the whole thing, but just the jar files that changed.

I only have one interface at the moment, but my Game class was getting too filled up with things, so I extracted the humanTurn() and all the methods that it called to a separate class. Since I also have to add a computer turn, this sounded like a good idea. The computer turn can be in a similar class with similar functions. Maybe I could extract the similarities to an interface? as I learnt to do in the TDD by Example book.
