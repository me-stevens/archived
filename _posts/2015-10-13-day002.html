In other topics, today I started reading "Test Driven Development by example", by Kent Beck. It's one of the stories I have to finish, and Jim told me to create a repo for that.

Test Driven Development, or "TDD", is something new for me. I have tested my code in the past, but never in this way. The first time I made tests as grown up people do them, it was for my Tic Tac Toe. Now I have to learn to write the test first, and the code last. Also, I have to learn to stop thinking about the big picture, because when you write code the TDD way, you have to start with very simple things. For example, a function whose only purpose is to return 5; and that's all it does, because you wrote it to make a test pass, and this test checks that the result of your function is 5.

So this is what I refer to when I say that "I've been programming my whole life, but I've been doing it wrong". This way of working blew my mind. When I paired with Mateu for the FizzBuzz in ruby, he told me that you have to write your tests in a way that you trust that the code is working for every possible case, without feeling insecure about it. For example, I would read an if-else statement and go through it thinking what happens in every case. Now, with TDD, I have to learn to stop doing that, and just trust the test.

The first thing I learnt from the book is, the TDD mantra:

RED - GREEN - REFACTOR

In other words:

  • RED: you first write a test that fails,
  • GREEN or, "make it work": then you make it pass by writing the code that works in the simplest and fastest way, even if it is in the most ugly way like, return 5;
  • REFACTOR or, "make it pretty": at this point is where you fix duplication, etc.

This is so similar to what I used to do in web development, which was where I first heard about the concepts of "best practices" and "testing" at all.

The objective you follow with this mantra is, to write clean code that works.
