Today I've been watching more Uncle Bob's videos. This is what I learnt from one of them:

  • Reduce local variable assignments, inline whenever you can.
  • Keep the number of arguments of your methods as low as possible, maximum three.
  • It's better to have setters and getters than a constructor full of variables.
  • Never pass a boolean as an argument, because it means that your method does two things. Better create two methods.
  • Passing by reference (output variables used as arguments) is confusing (I can think of "int &index" in C or "out int index" in C#).
  • If you can pass a null, it means your tests are not preventing you from using nulls.
  • Put the public members and methods at the top, the private ones at the bottom.

Regarding switch statements: We can get rid of them using the Dependency Inversion Principle, which opposes the flow of control.

If A depends on B, they follow the flow of control, and they have to be deployed together. If we add an interface in between, the source code dependency opposes the flow of control. Many B's can be plug to A using polymorphism, and we can have independent deployability.

The other option is to place them where they make no harm, for example, in the main part of your application's code.

Polymorphism in my case seems like an overkill, since I would have to create a new class everytime I want to add an option to the menu. Moving it to the main part of my app would mean moving it to either the GameRunner class or the Game, where it was before, which are also not good options. I DON'T KNOW WHAT TO DO.

Finally, he also speaks about the Law of Demeter, or "tell, don't ask", which means that functions should not know the navigation of your system (which is very common in Java). For example, don't do this:

getX().getY().getZ().doSomething()

The Law of Demeter is sometimes called "The suggestion of Demeter", because it's very hard to follow.
